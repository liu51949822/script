## 一、通用配置步骤

### Secrets 配置

在仓库 `Settings > Secrets and variables > Actions > Secrets` 中按需添加：

| Secret | 键名 | 工作流 | 获取方式 |
|--------|------|--------|----------|
| Docker Hub 用户名 | `DOCKERHUB_USERNAME` | docker-build-push | 注册 Docker Hub 账号 |
| Docker Hub Token | `DOCKERHUB_TOKEN` | docker-build-push | Docker Hub > Account Settings > Security > Access Tokens |
| SSH 服务器地址 | `HOST` | deploy-ssh | 你的服务器 IP 或域名 |
| SSH 用户名 | `USERNAME` | deploy-ssh | 如 `root` 或 `deploy` |
| SSH 私钥 | `KEY` | deploy-ssh | `cat ~/.ssh/id_rsa \| pbcopy` |
| SSH 端口 | `PORT` | deploy-ssh | 通常 `22`，修改了则填实际端口 |
| Gitee 镜像 URL | `GITEE_MIRROR_URL` | sync-mirror | 如 `https://gitee.com/yourname/repo.git` |
| Vercel Token | `VERCEL_TOKEN` | nextjs-ci-cd | Vercel > Settings > Tokens |
| Vercel Org ID | `VERCEL_ORG_ID` | nextjs-ci-cd | Vercel > Settings > General |
| Vercel Project ID | `VERCEL_PROJECT_ID` | nextjs-ci-cd | Vercel > Project > Settings > General |

### 启用 Actions

1. 进入仓库 `Settings > Actions > General`
2. 确保 `Allow all actions and reusable workflows` 已勾选
3. `Workflow permissions` 设为 `Read and write permissions`

### 手动触发

大部分工作流支持 `workflow_dispatch`，在仓库 Actions 页面点击工作流 → `Run workflow` 即可。

## 二、各工作流详细说明

### 1. 🐳 docker-build-push — Docker 镜像构建与推送

**文件**: `docker-build-push.yml`

**功能**: 当 `docker/` 目录下的 compose 文件变更时，自动检测变更的服务，构建 Docker 镜像并推送至 Docker Hub。

**触发条件**:
| 事件 | 条件 |
|------|------|
| push | main 分支，`docker/**` 路径变更 |
| workflow_dispatch | 手动触发，可选指定服务名 |

**配置步骤**:
1. 在仓库 Secrets 中添加 `DOCKERHUB_USERNAME` 和 `DOCKERHUB_TOKEN`
2. 修改 `docker-build-push.yml` 中 `matrix.service` 列表为你的实际服务名
3. 修改 `IMAGE_PREFIX` 为你的 Docker Hub 用户名

**自定义**:
```yaml
# 修改为你需要构建的服务列表
strategy:
  matrix:
    service: [ "pgsql", "mysql", "redis", "nginx", "mongodb", "elasticsearch" ]

# 修改镜像前缀
env:
  IMAGE_PREFIX: yourhubusername
```

### 2. 🔍 docker-compose-validate — Docker Compose 文件验证

**文件**: `docker-compose-validate.yml`

**功能**: 自动验证所有 `installcompose.file` 的 YAML 语法、必需字段（services/networks），检测端口冲突。

**触发条件**:
| 事件 | 条件 |
|------|------|
| pull_request | 目标 main，`docker/**` 变更 |
| push | main 分支，`docker/**` 变更 |
| workflow_dispatch | 手动触发 |

**输出**: 每个文件验证结果 + 端口冲突报告

**文件**: `repo-readme-sync.yml`

**功能**: compose 文件变更时自动运行 `python/index-generator.py` 更新 `docker/README.md`。

**触发条件**:
| 事件 | 条件 |
|------|------|
| push | main 分支，docker/ 文件变更 |
| workflow_dispatch | 手动触发 |

**前置条件**: 需要 `python/index-generator.py` + `pip install pyyaml`

### 4. 🚀 auto-release — 自动发布 Release

**文件**: `auto-release.yml`

**功能**: 推送 `v*` 格式 tag 时自动创建 GitHub Release，自动生成 Changelog。

**触发条件**: 推送 tag（如 `v1.0.0`、`v2.0.0-beta`）

**使用方法**:
```bash
git tag v1.0.0
git push origin v1.0.0
# Release 会自动创建在 https://github.com/你的仓库/releases
```

**版本规则**:
- `v1.0.0` → 正式版 Release
- `v1.0.0-rc1` → Pre-release
- `v1.0.0-beta` → Pre-release
- `v1.0.0-alpha` → Pre-release

### 5. 📤 deploy-ssh — SSH 远程部署

**文件**: `deploy-ssh.yml`

**功能**: 通过 rsync 同步仓库文件到远程服务器，然后通过 SSH 执行 `compose-up.sh` 重启 Docker 服务。

**触发条件**:
| 事件 | 条件 |
|------|------|
| push | main 分支 |
| workflow_dispatch | 手动触发，可选 production/staging 环境 |

**前置条件**:
- 远程服务器已安装 Docker 和 rsync
- 目标路径 `/opt/scripts` 存在（或修改为目标路径）

**配置 Secrets**: `HOST`, `USERNAME`, `KEY`, `PORT`

**修改部署路径**:
```yaml
# 在 deploy-ssh.yml 中修改
SOURCE: "."
TARGET: "/opt/scripts"        # ← 改为你的实际路径
script: |
  cd /opt/scripts              # ← 对应修改
  bash base/compose-up.sh --all
```

### 6. ⚛️ nextjs-ci-cd — Next.js CI/CD

**文件**: `nextjs-ci-cd.yml`

**功能**: Next.js 项目的完整 CI/CD 流程 — 安装依赖、Lint、测试、构建，然后部署到 Vercel。

**触发条件**:
| 事件 | 条件 |
|------|------|
| push | main 或 production 分支 |
| pull_request | 目标 main |

**配置步骤**:
1. 添加 Secrets: `VERCEL_TOKEN`, `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID`
2. 修改 `.env.test` 中的环境变量为实际值
3. 如用 pnpm，确保 `package.json` 中有 `lint`, `test`, `build` 脚本

### 7. 🕐 stale-issues — Issue 自动清理

**文件**: `stale-issues.yml`

**功能**: 自动标记和关闭长时间无活动的 Issue 和 PR。

**触发条件**:
| 事件 | 条件 |
|------|------|
| schedule | 每天 UTC 0:00 |
| workflow_dispatch | 手动触发 |

**配置参数**:
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `days-before-issue-stale` | 60 | 60 天无活动标记为 stale |
| `days-before-issue-close` | 14 | stale 后 14 天自动关闭 |
| `days-before-pr-stale` | 90 | PR 90 天无更新标记 stale |
| `exempt-issue-labels` | pinned,security,bug | 这些标签的 Issue 不会被关闭 |

### 8. 🔄 sync-mirror — 仓库镜像同步

**文件**: `sync-mirror.yml`

**功能**: 自动将仓库推送到 Gitee 镜像仓库。

**触发条件**:
| 事件 | 条件 |
|------|------|
| push | main 分支 |
| schedule | 每 6 小时 |
| workflow_dispatch | 手动触发 |

**配置步骤**:
1. 在 Gitee 创建同名仓库（或任意名称）
2. 在 GitHub Secrets 添加 `GITEE_MIRROR_URL`
   - 格式：`https://gitee.com/你的用户名/仓库名.git`
3. 添加 Gitee 用户名和密码作为 Secrets（如需要认证）

**同时同步到 GitLab**:
```yaml
# 在 sync-mirror.yml 中添加
- name: 🔄 同步到 GitLab
  run: |
    git remote add gitlab ${{ secrets.GITLAB_MIRROR_URL }} 2>/dev/null || true
    git push gitlab main:master --force --tags
```

### 9. 📦 backup-schedule — 定时备份

**文件**: `backup-schedule.yml`

**功能**: 每周打包仓库内容（不含 `.git` 和数据目录），上传到 Release 作为备份。

**触发条件**:
| 事件 | 条件 |
|------|------|
| schedule | 每周日 UTC 4:00 |
| workflow_dispatch | 手动触发 |

**备份内容**: `docker/`, `base/`, `python/`, `gitaction/`, `mkfile.mk`, `.gitignore`

**排除内容**: `.git`, `docker/*/*-data`, `docker/*/*-db`, `backups/`

**保留策略**: 自动保留最近 5 个备份 Release

**修改 cron**:
```yaml
# 每天凌晨4点备份
- cron: '0 4 * * *'

# 每月1号备份
- cron: '0 4 1 * *'

# 每周一三五备份
- cron: '0 4 * * 1,3,5'
```

### 10. 🔒 security-audit — 安全审计

**文件**: `security-audit.yml`

**功能**: 多重安全检查 — Gitleaks 扫描密钥泄露、CodeQL 代码分析、compose 文件密码检查。

**触发条件**:
| 事件 | 条件 |
|------|------|
| push | main 分支 |
| pull_request | 目标 main |
| schedule | 每周一 UTC 2:00 |
| workflow_dispatch | 手动触发 |

**三个检查阶段**:
1. **Gitleaks**: 扫描 Git 历史中的敏感信息（API Key、Token 等）
2. **密码检查**: 统计包含占位密码的文件数量，提醒替换
3. **CodeQL**: 静态代码安全分析（Python, JavaScript）

---

### 11. 🏷️ auto-labeler — PR 自动标签

**文件**: `auto-labeler.yml`

**功能**: 根据 PR 修改的文件路径自动添加标签。

**触发条件**: PR 创建或更新时

**配置方式**: 创建 `.github/labeler.yml` 定义标签规则

```yaml
# 示例: .github/labeler.yml
docker:
  - changed-files:
      - any-glob-to-any-file: 'docker/**'

base:
  - changed-files:
      - any-glob-to-any-file: 'base/**'

python:
  - changed-files:
      - any-glob-to-any-file: 'python/**'

github-actions:
  - changed-files:
      - any-glob-to-any-file: '.github/workflows/**'

documentation:
  - changed-files:
      - any-glob-to-any-file: '**/*.md'
```

## 三、快速部署

### 一次性部署所有工作流

```bash
# 已在此仓库部署，无需重复操作
# 以下为你的其他仓库使用方式:

# 方法一：直接复制
cp gitaction/*.yml /path/to/other-repo/.github/workflows/

# 方法二：从本仓库同步
git clone https://github.com/liu51949822/script.git
cp script/gitaction/*.yml your-repo/.github/workflows/
```

### 验证工作流语法

```bash
# 安装 actionlint
brew install actionlint

# 验证所有工作流
actionlint .github/workflows/*.yml
actionlint gitaction/*.yml

# 检查结果
echo $?  # 0 = 全部通过
```

## 四、常见问题

| 问题 | 解决方案 |
|------|----------|
| **Actions 不触发** | 检查文件是否在 `.github/workflows/` 目录下，且扩展名为 `.yml` |
| **权限不足** | Settings > Actions > General > Workflow permissions 设为 Read and write |
| **Secret 未生效** | Secrets 区分环境，确保在正确的 environment 下添加 |
| **SSH 连接失败** | 检查 `HOST`、`USERNAME`、`KEY`、`PORT` 是否正确，服务器防火墙是否开放 |
| **Docker 推送 401** | Docker Hub Token 需要有 Read & Write 权限 |
| **备份 Release 过多** | 工作流会自动清理，保留最近 5 个 |
| **Gitee 同步失败** | 确保 Gitee 仓库已创建且 URL 正确 |
